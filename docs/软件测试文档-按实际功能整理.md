## 1. 测试内容概述

### 1.1 项目背景

本系统是基于 Python Flask + Supabase PostgreSQL 的校园二手交易平台，实现 B/S 架构的 Web 应用。系统包含 7 大功能模块，共 50 个功能需求，涵盖用户账号、商品发布、浏览收藏、订单交易、消息评价、通知和管理后台等核心业务。

### 1.2 测试目标

系统测试的目的是验证系统是否满足需求规格的定义，找出与需求规格不符和矛盾的地方，从而提出更加完善的方案。具体目标包括：

- 验证系统各功能模块是否正确实现
- 确保系统在各种边界条件下稳定运行
- 发现潜在的性能瓶颈和安全漏洞
- 保证系统数据的完整性和一致性

### 1.3 测试范围

本次测试主要覆盖以下内容：

| 模块 | 需求编号 | 功能 |
|------|---------|------|
| A 用户账号 | F01-F06 | 注册、登录（5次锁定）、登出、找回密码、修改密码、编辑资料 |
| B 商品发布 | F07-F16 | 发布商品、上传图片、分类选择、定价、草稿、管理、编辑、删除、上下架 |
| C 浏览收藏 | F17-F23 | 首页推荐、分类浏览、搜索筛选、商品详情、收藏管理 |
| D 订单交易 | F24-F33 | 提交订单、线上/线下交易、订单管理、确认、取消、拒绝、模拟支付、确认收货 |
| E 消息评价 | F34-F40 | 站内消息、聊天、双向评价、举报 |
| F 通知 | F41-F42 | 系统通知、订单通知 |
| G 管理后台 | F43-F50 | 管理员登录、用户管理、商品审核、分类管理、举报处理、数据统计 |

## 2. 软件测试过程

### 2.1 测试流程

```
用户立项 → 系统需求 → 软件需求 → 软件设计 → 软件编码
                    ↓              ↓            ↓
                  系统测试 ← 确认测试 ← 集成测试 ← 单元测试
                                        ↓
                                    交付用户
```

### 2.2 测试阶段划分

| 测试阶段 | 测试对象 | 测试目的 | 主要技术 |
|---------|---------|---------|---------|
| 单元测试 | 软件模块 | 验证单个模块功能正确性 | 白盒测试 |
| 集成测试 | 模块组合 | 验证模块间接口和协作 | 灰盒测试 |
| 确认测试 | 完整系统 | 验证是否满足用户需求 | 黑盒测试 |
| 系统测试 | 整个产品 | 验证系统整体功能和性能 | 综合测试 |

## 3. 测试计划及测试用例

### 3.1 制定测试计划

#### 3.1.1 测试范围

- **功能测试**：覆盖所有 50 个功能需求
- **性能测试**：验证系统在高并发下的响应能力
- **安全测试**：验证系统的安全性和数据保护能力
- **兼容性测试**：验证系统在不同浏览器和设备上的兼容性

#### 3.1.2 测试方法

##### 3.1.2.1 等价类划分

等价类划分是将输入数据划分为若干等价类，从每个等价类中选取代表性数据作为测试用例，以减少测试用例数量。

##### 3.1.2.2 边界值分析

边界值分析是对输入或输出的边界值进行测试，因为边界条件往往是错误的高发区。

##### 3.1.2.3 场景测试法

场景测试法是模拟用户真实使用流程，从用户进入网站到完成某个目标的完整路径。

##### 3.1.2.4 错误推测法

错误推测法是根据经验推测用户可能遇到的问题，例如网络延迟时提交订单、用户刷新页面造成重复提交等。

#### 3.1.3 测试工具

| 工具 | 用途 |
|------|------|
| pytest | 单元测试框架 |
| Selenium | 自动化测试工具 |
| Postman | API 接口测试 |
| JMeter | 性能测试 |

#### 3.1.4 测试完成准则

1. **功能测试完成**：所有功能测试用例执行完毕，通过率 ≥ 95%
2. **性能测试完成**：系统响应时间 ≤ 2 秒，并发用户数 ≥ 100
3. **安全测试完成**：无高危安全漏洞
4. **兼容性测试完成**：在所有支持的浏览器上正常运行
5. **回归测试完成**：修复的缺陷无回归问题
