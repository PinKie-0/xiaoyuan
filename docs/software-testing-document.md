## 1. 测试内容概述

### 1.1 项目背景

本系统是基于 Python Flask + Supabase PostgreSQL 的校园二手交易平台，实现 B/S 架构的 Web 应用。系统包含 7 大功能模块，共 50 个功能需求，涵盖用户账号、商品发布、浏览收藏、订单交易、消息评价、通知和管理后台等核心业务。

### 1.2 测试目标

系统测试的目的是验证系统是否满足需求规格的定义，找出与需求规格不符和矛盾的地方，从而提出更加完善的方案。具体目标包括：

- 验证系统各功能模块是否正确实现
- 确保系统在各种边界条件下稳定运行
- 发现潜在的性能瓶颈和安全漏洞
- 保证系统数据的完整性和一致性

### 1.3 测试范围

本次测试主要覆盖以下内容：

| 模块 | 需求编号 | 功能 |
|------|---------|------|
| A 用户账号 | F01-F06 | 注册、登录（5次锁定）、登出、找回密码、修改密码、编辑资料 |
| B 商品发布 | F07-F16 | 发布商品、上传图片、分类选择、定价、草稿、管理、编辑、删除、上下架 |
| C 浏览收藏 | F17-F23 | 首页推荐、分类浏览、搜索筛选、商品详情、收藏管理 |
| D 订单交易 | F24-F33 | 提交订单、线上/线下交易、订单管理、确认、取消、拒绝、模拟支付、确认收货 |
| E 消息评价 | F34-F40 | 站内消息、聊天、双向评价、举报 |
| F 通知 | F41-F42 | 系统通知、订单通知 |
| G 管理后台 | F43-F50 | 管理员登录、用户管理、商品审核、分类管理、举报处理、数据统计 |

## 2. 软件测试过程

### 2.1 测试流程

```
用户立项 → 系统需求 → 软件需求 → 软件设计 → 软件编码
                    ↓              ↓            ↓
                  系统测试 ← 确认测试 ← 集成测试 ← 单元测试
                                      ↓
                                  交付用户

### 2.2 测试阶段划分

| 测试阶段 | 测试对象 | 测试目的 | 主要技术 |
|---------|---------|---------|---------|

## 3. 测试计划

### 3.1 测试范围

- **功能测试**：覆盖所有 50 个功能需求
- **性能测试**：验证系统在高并发下的响应能力
- **安全测试**：验证系统的安全性和数据保护能力
- **兼容性测试**：验证系统在不同浏览器和设备上的兼容性

### 3.2 测试方法

#### 3.2.1 黑盒测试

| 特点 | 说明 |
|------|------|
| 主要应用 | 功能测试 |
| 核心思想 | 不基于系统内部设计和实现 |
| 用例设计依据 | 功能定义和需求说明书 |
| 关注点 | 测试数据选择和测试结果分析 |
| 优点 | 能确保从用户角度出发进行测试 |
| 缺点 | 对内部实现的 bug 不容易发现；不能提供直观的测试覆盖率 |
| 主要方法 | 等价类划分、边界值分析、因果图、决策表测试 |

#### 3.2.2 白盒测试

| 特点 | 说明 |
|------|------|
| 主要应用 | 结构测试 |
| 核心思想 | 需要了解系统的整体设计和实现 |
| 用例设计依据 | 源代码审查 |
| 关注点 | 系统的控制流和数据流 |
| 优点 | 能对程序内部的特定部位进行覆盖测试 |
| 缺点 | 不能确保系统是否完全符合需求；代价大；需源代码完成后才能进行 |
| 主要方法 | 独立路径测试、逻辑判断测试、数据结构测试、覆盖率测试 |

#### 3.2.3 测试方法对比

| 比较项目 | 黑盒测试 | 白盒测试 |
|---------|---------|---------|
| 主要应用 | 功能测试 | 结构测试 |
| 测试依据 | 需求规格说明书 | 源代码 |
| 测试时机 | 可在开发早期进行 | 需代码完成后 |
| 测试成本 | 较低 | 较高 |
| 发现问题类型 | 功能缺陷、业务逻辑错误 | 代码缺陷、逻辑漏洞 |
| 测试覆盖率 | 难以量化 | 可精确度量 |

### 4.4 白盒测试方法

白盒测试是基于程序内部结构和逻辑的测试方法，主要包括路径覆盖、条件覆盖、语句覆盖等。

#### 4.4.1 submit_review 函数路径覆盖分析

根据 F38 买家评价卖家功能的伪代码，`submit_review` 函数的控制流如下：

```
开始
  ↓
获取订单（order_id, reviewer_id, score, content）
  ↓
订单存在？
  ├─ 否 → 返回(失败, '订单不存在。') → 结束
  └─ 是 → 继续
  ↓
订单状态为COMPLETED？
  ├─ 否 → 返回(失败, '只能评价已完成的订单。') → 结束
  └─ 是 → 继续
  ↓
评价人有权限？(reviewer_id in [buyer_id, seller_id])
  ├─ 否 → 返回(失败, '无权评价此订单。') → 结束
  └─ 是 → 继续
  ↓
已评价过？(existing review found)
  ├─ 是 → 返回(失败, '您已经评价过此订单。') → 结束
  └─ 否 → 继续
  ↓
评分在1~5之间？
  ├─ 否 → 返回(失败, '评分必须在1~5之间。') → 结束
  └─ 是 → 继续
  ↓
过滤评价内容
  ↓
创建评价记录
  ↓

  ↓

### 7.4 测试结论

根据测试执行结果，本系统整体测试通过率为 **95.2%**，达到了预设的测试完成准则（≥95%）。系统的核心功能模块（用户账号、浏览收藏、消息评价、通知）测试通过率为100%，功能实现较为完善。

主要发现的问题包括：
1. **安全性问题**：搜索框存在SQL注入漏洞，需要立即修复
2. **数据一致性问题**：订单确认后商品状态未同步更新
3. **用户体验问题**：大文件上传时页面响应延迟
4. **数据完整性问题**：分类删除未检查商品引用

整体而言，系统已具备上线条件，但需要先修复严重级别的SQL注入漏洞和高级别的数据一致性问题。

### 7.5 改进建议

#### 7.5.1 立即修复（严重/高优先级）

1. **修复SQL注入漏洞**（DEF-004）
   - 修改搜索接口的SQL查询语句，使用参数化查询
   - 对用户输入进行严格的字符过滤和转义处理
   - 添加输入验证层，禁止特殊字符输入

2. **修复订单确认后商品状态不同步问题**（DEF-002）
   - 在`confirm_order`函数中添加商品状态更新逻辑
   - 使用数据库事务确保订单和商品状态的原子性更新

#### 7.5.2 计划修复（中/低优先级）

3. **优化大文件上传体验**（DEF-001）
   - 添加前端文件大小预检查，在选择文件后立即验证
   - 实现文件上传进度条，提升用户体验

4. **完善分类删除逻辑**（DEF-003）
   - 删除分类前检查是否有商品引用
   - 提供批量转移商品到其他分类的功能

#### 7.5.3 建议增强

5. **添加输入验证层**
   - 统一前端和后端的输入验证规则
   - 使用正则表达式对所有输入进行格式校验

6. **增加自动化测试覆盖率**
   - 为核心业务流程编写集成测试用例
   - 实现API接口的自动化回归测试

7. **完善日志记录**
   - 添加操作日志记录，便于问题追踪和审计
   - 实现错误日志的集中管理和告警机制
